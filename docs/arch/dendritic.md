# Dendritic Architecture Pattern

The "Dendritic" pattern in this repository is a filesystem-driven modularity strategy. It uses automatic discovery to branch system configurations from a central root, similar to the branching structure of a dendrite.

## Current Implementation

The architecture relies on three core technologies:
1. **`import-tree`**: Recursively scans the `modules/` directory and converts the filesystem structure into a Nix attribute set.
2. **`flake-parts`**: Provides the framework for defining flake outputs (modules, configurations) in a modular way.
3. **Recursive Collector**: A function in `flake.nix` that traverses the tree generated by `import-tree`, identifies valid modules, and flattens them for use in `mkFlake`.

### Strengths
- **Zero-Boilerplate Discovery**: Adding a new file to `modules/` automatically makes it available to the flake.
- **Strict Hierarchy**: The filesystem structure directly maps to the configuration logic.

## The Portability Problem

Currently, the implementation is **NixOS-locked**. While the pattern *should* allow for cross-platform use (NixOS, Darwin, Standalone Home Manager), the current structure prevents this for several reasons:

### 1. Hardcoded Output Namespaces
Most modules are defined directly under the `flake.nixosModules` attribute.
```nix
flake.nixosModules.dev-cli = { ... };
```
This forces any host using these modules to be a NixOS system. A Darwin system or a standalone Home Manager installation cannot use these attributes.

### 2. Tight Coupling of Logic
System-level configuration (NixOS options like `environment.systemPackages`) is mixed with user-level configuration (`home-manager` options).
```nix
flake.nixosModules.example = { pkgs, ... }: {
  environment.systemPackages = [ ... ]; # NixOS specific
  home-manager.users.crow = { ... };    # Portable logic trapped in NixOS module
};
```

### 3. Hardcoded User Identities
Modules explicitly define configurations for the user `crow`. This prevents the "branches" of the dendrite from being used by other users or on systems where the primary user is named differently.

## Future Evolution: True Dendritic Modularity

To achieve the ideal "graftable" architecture, the following refactoring is required:

1. **Separate Logic from Wrapper**:
   - Extract the core logic (packages, dotfiles, shell aliases) into generic attribute sets or "shared" modules.
   - Create specific wrappers for `nixosModules`, `darwinModules`, and `homeModules` that import the shared logic.

2. **Parameterize the System**:
   - Use options to define the username and other system-specific variables instead of hardcoding them within the module.

3. **Multi-Output Discovery**:
   - Update the collector in `flake.nix` to handle different types of outputs, allowing a single file in `modules/` to export multiple flavors of the same module (e.g., a NixOS version and a Home Manager version).

By decoupling the *what* (configuration logic) from the *how* (system-specific implementation), the repository can truly function as a dendritic system where logic is shared across any Nix-supported platform.
